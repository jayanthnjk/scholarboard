# Auth RBAC & Tenant Fix - Bugfix Design

## Overview

The multi-tenant SaaS ERP education platform has critical defects preventing production-grade RBAC enforcement. The sidebar component (`src/components/layout/sidebar.tsx`) renders a hardcoded static `SECTIONS` constant instead of consuming the dynamic navigation generated by `ConfigProvider`. Post-login routing sends Students and Parents to `/dashboard` instead of their respective portal pages. The `RouteGuard` component in `App.tsx` is used without any permission props, making all module routes accessible to any authenticated user. Super Admin console pages exist but are unreachable via navigation. The fee module lacks action-level permission gating. No role/tenant context is displayed in the UI.

The fix strategy is to wire the existing RBAC infrastructure (ConfigProvider navigation generation, RouteGuard permission checking, usePermission hook) into the UI layer where it is currently disconnected.

## Glossary

- **Bug_Condition (C)**: Any authenticated session where the UI fails to enforce RBAC — sidebar shows hardcoded navigation, routes lack permission guards, or role context is missing
- **Property (P)**: The desired behavior where sidebar renders ConfigProvider-generated navigation, routes enforce module/action permissions, and role/tenant context is visible
- **Preservation**: Existing School Admin functionality, sidebar collapse/expand, logout flow, demo accounts, session management, command palette, and existing usePermission checks on StudentListPage/StudentProfilePage must remain unchanged
- **ConfigProvider**: The provider at `src/providers/ConfigProvider.tsx` that generates filtered `NavigationConfig` based on tenant modules, user role, permissions, and feature flags
- **RouteGuard**: The component at `src/components/RouteGuard.tsx` that supports `resource`, `action`, `module`, `page`, and `roles` props but is currently invoked without any permission props in `App.tsx`
- **ROLE_DASHBOARD_MAP**: The mapping in `AuthProvider.tsx` that determines post-login redirect path per role
- **usePermission**: The hook that provides `hasPermission(resource, action)`, `hasModuleAccess(module)`, and `hasPageAccess(page)` checks
- **SUPER_ADMIN_NAVIGATION_ITEMS**: Navigation items defined in ConfigProvider for the Super Admin role, currently not consumed by the sidebar
- **BASE_NAVIGATION_ITEMS**: Navigation items defined in ConfigProvider for school-level roles, filtered by tenant modules and feature flags

## Bug Details

### Bug Condition

The bug manifests for ALL authenticated users because the sidebar never consumes the ConfigProvider's dynamically generated navigation. Additionally, Students and Parents are routed to the wrong landing page, no route-level RBAC guards exist on module paths, Super Admin pages are inaccessible via navigation, fee module pages lack action-level permission gating, and no role/tenant context is displayed.

**Formal Specification:**
```
FUNCTION isBugCondition(input)
  INPUT: input of type AuthenticatedSession { role: UserRole, permissions: Permission[], tenantId: string, targetRoute: string }
  OUTPUT: boolean

  // Sub-condition 1: Navigation bug — sidebar is hardcoded for ALL roles
  LET navBug = true  // Always true — sidebar never uses ConfigProvider

  // Sub-condition 2: Routing bug — Students/Parents land on wrong page
  LET routingBug = input.role IN ['student', 'parent']

  // Sub-condition 3: Access control bug — any restricted user can access unauthorized routes via URL
  LET accessBug = NOT rbacEngine.hasModuleAccess(input.role, routeToModule(input.targetRoute))
                  AND input.targetRoute IS accessed

  // Sub-condition 4: Action-level bug — fee module buttons not gated
  LET actionBug = input.targetRoute.startsWith('/fees')
                  AND actionButtonsRenderedWithoutPermissionCheck()

  // Sub-condition 5: Context display bug — no role/tenant info shown
  LET contextBug = true  // Always true — sidebar never shows role badge or tenant name

  RETURN navBug OR routingBug OR accessBug OR actionBug OR contextBug
END FUNCTION
```

### Examples

- **Super Admin login**: Routes to `/admin` correctly, but sidebar shows Dashboard/Analytics/Calendar/Messages/Registration/Students/Library/Courses/Faculty/Exam Board/Financial Record/Help/Settings instead of Tenants/Platform Analytics/Subscriptions/School Dashboard/Students/Fees/Reports/Settings
- **Student login**: Routes to `/dashboard` (incorrect — should be `/student-portal`), sidebar shows all admin modules instead of student portal navigation
- **Parent login**: Routes to `/dashboard` (incorrect — should be `/parent-portal`), sidebar shows all admin modules instead of parent portal navigation
- **Teacher navigating to `/fees/structure`**: Page renders because RouteGuard has no `resource="fees" action="create"` prop — Teacher should be blocked
- **Accountant viewing sidebar**: Sees Library, Courses, Staff, Exams, Admissions, Calendar — modules outside their permission scope
- **Any user viewing sidebar header**: Shows hardcoded "Sunrise Academy" and "Academic Year 2024-25" with no role badge or user name display

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- School Admin login continues to route to `/dashboard` with access to all institution-level modules
- Sidebar collapse/expand toggle with smooth CSS transitions and icon-only mode continues to work
- Logout button continues to call `sessionManager.logout()`, clear session, and redirect to `/login`
- Demo accounts quick-fill feature on login page continues to work for all 6 roles
- Session expiry redirect to `/login` with URL preservation continues to work
- MSW mock server authentication against mockUsers array continues to return correct user objects
- All existing module pages (StudentListPage, StudentProfilePage, AdmissionFormPage, all fee pages, DashboardPage, TenantManagementPage, PlatformAnalyticsPage, SubscriptionPage, scaffolded pages) continue to render with current functionality
- Command palette (Ctrl+K / Cmd+K) continues to function
- Offline banner continues to display on network disconnection
- Theme preferences continue to persist via ThemeProvider
- Route-level code splitting with React.lazy() and Skeleton fallbacks continues to work
- StudentListPage "Add Student" button continues to be hidden when user lacks `students.create` permission
- StudentProfilePage "Edit" button continues to be hidden when user lacks `students.edit` permission

**Scope:**
All inputs that do NOT involve sidebar rendering, post-login routing, route-level permission enforcement, fee module action gating, or role/tenant context display should be completely unaffected by this fix. This includes:
- Page content rendering within authorized module pages
- API call patterns and mock server responses
- Form submissions and data mutations
- CSS styling and responsive layout (except sidebar content changes)
- Third-party integrations and service worker behavior

## Hypothesized Root Cause

Based on the bug description and code analysis, the root causes are:

1. **Sidebar ignores ConfigProvider output**: The `Sidebar` component at `src/components/layout/sidebar.tsx` defines a static `SECTIONS` constant with hardcoded navigation items. Although `AppLayout` in `App.tsx` calls `useConfig()` and destructures `navigation`, it stores it in `_nav` (prefixed with underscore, intentionally unused). The sidebar accepts no `items` prop driven by ConfigProvider.

2. **Incorrect ROLE_DASHBOARD_MAP for Student/Parent**: In `AuthProvider.tsx`, the `ROLE_DASHBOARD_MAP` maps both `student` and `parent` to `/dashboard` instead of `/student-portal` and `/parent-portal` respectively. The portal routes exist in `App.tsx` but are never the landing page.

3. **RouteGuard used without permission props**: In `App.tsx`, all module routes are wrapped in a single `<RouteGuard>` with no `resource`, `module`, `page`, or `roles` props. The RouteGuard code shows that when no permission props are specified, it falls through to `return <>{children}</>` after authentication check only.

4. **Fee module pages lack usePermission checks**: Unlike `StudentListPage` which properly uses `hasPermission('students', 'create')` to gate the "Add Student" button, fee module pages render action buttons (Create Fee Structure, Collect Payment, etc.) without any permission checks.

5. **No role/tenant context in sidebar UI**: The sidebar header hardcodes "Sunrise Academy" and "Academic Year 2024-25" text. It receives `userName` as a prop but doesn't display it prominently with a role badge. There is no reading of tenant configuration state.

6. **ConfigProvider navigation lacks role-specific filtering for restricted roles**: The `BASE_NAVIGATION_ITEMS` in ConfigProvider have empty `permissions: []` arrays, meaning the permission-based filtering in `filterItems` never excludes items based on role permissions — only feature flags are checked. Role-specific navigation filtering needs to be implemented.

## Correctness Properties

Property 1: Bug Condition - Sidebar Renders ConfigProvider Navigation

_For any_ authenticated session where `ConfigProvider.navigation.items` generates a filtered list of navigation items based on the user's role, permissions, and tenant modules, the sidebar component SHALL render exactly those items (and no hardcoded items), ensuring only permitted modules are visible in the navigation.

**Validates: Requirements 2.1, 2.2, 2.3, 2.4, 2.5, 2.6, 2.7, 2.17**

Property 2: Bug Condition - Post-Login Routing Matches Role

_For any_ authenticated session where the user's role is `student`, the system SHALL route to `/student-portal`, and where the role is `parent`, the system SHALL route to `/parent-portal`, ensuring each role lands on their appropriate dashboard.

**Validates: Requirements 2.4, 2.5**

Property 3: Bug Condition - Route-Level RBAC Enforcement

_For any_ authenticated session where a user navigates to a module route they lack permissions for (determined by `rbacEngine.hasModuleAccess`), the RouteGuard SHALL redirect to `/permission-denied` instead of rendering the page content.

**Validates: Requirements 2.10, 2.14, 2.22**

Property 4: Bug Condition - Fee Module Action-Level Gating

_For any_ authenticated session accessing fee module pages, action buttons (Create Fee Structure, Collect Payment, Process Refund, Generate Receipt) SHALL only render when the user has the corresponding permission (`fees.create`, `fees.collect`, `fees.refund`, `fees.delete`).

**Validates: Requirements 2.21, 2.23**

Property 5: Bug Condition - Super Admin Console Accessible

_For any_ authenticated session with `super_admin` role, the sidebar navigation SHALL include links to `/admin/tenants`, `/admin/analytics`, and `/admin/subscriptions`, making all platform management pages reachable.

**Validates: Requirements 2.1, 2.8, 2.11, 2.12**

Property 6: Bug Condition - Role/Tenant Context Display

_For any_ authenticated session, the sidebar SHALL display the tenant name from tenant configuration state (not hardcoded), the user's display name, and a color-coded role badge indicating the active role.

**Validates: Requirements 2.9, 2.13, 2.18**

Property 7: Preservation - Existing Functionality Unchanged

_For any_ input where the bug condition does NOT hold (School Admin accessing authorized modules, existing usePermission checks on Student pages, logout flow, sidebar collapse, command palette, demo accounts, session management), the fixed code SHALL produce exactly the same behavior as the original code, preserving all existing functionality.

**Validates: Requirements 3.1, 3.2, 3.3, 3.4, 3.5, 3.6, 3.7, 3.8, 3.9, 3.10, 3.11, 3.12, 3.13, 3.14**

## Fix Implementation

### Changes Required

Assuming our root cause analysis is correct:

**File**: `src/components/layout/sidebar.tsx`

**Changes**:
1. **Remove hardcoded SECTIONS constant**: Delete the static navigation structure
2. **Accept navigation items from props**: Add a `navigationItems` prop (type `NavigationItem[]`) driven by ConfigProvider output
3. **Render dynamic navigation**: Map over `navigationItems` to render sidebar items with icons resolved from a lookup map
4. **Add role/tenant context to header**: Replace hardcoded "Sunrise Academy" with `tenantName` prop, add `userRole` prop for role badge, display `userName` prominently
5. **Add role badge with color coding**: Super Admin = red, School Admin = blue, Accountant = green, Teacher = purple, Staff = gray, Student = teal, Parent = orange
6. **Add Super Admin mode indicator**: When path starts with `/admin/`, show "Platform Admin" badge in header

**File**: `src/App.tsx` (AppLayout function)

**Changes**:
1. **Pass ConfigProvider navigation to Sidebar**: Use `useConfig().navigation.items` instead of unused `_nav`, pass as `navigationItems` prop to Sidebar
2. **Pass tenant context to Sidebar**: Read tenant name from `useAppStore` and pass as `tenantName` prop
3. **Pass role info to Sidebar**: Pass `user.role` as prop for role badge rendering
4. **Add per-module RouteGuard wrappers**: Wrap each module route group with `<RouteGuard>` including appropriate permission props:
   - `/admin/*` → `roles={['super_admin']}`
   - `/fees` → `module="fees"`
   - `/fees/structure` → `resource="fees" action="create"`
   - `/fees/make-payment` → `resource="fees" action="collect"`
   - `/staff` → `module="hr"`
   - `/settings` → `resource="settings" action="view"`
   - `/students/admission` → `resource="students" action="create"`
   - `/admissions` → `module="admissions"`
   - `/transport` → `module="transport"`
   - `/library` → `module="library"`
   - `/exams` → `module="exams"`
   - `/attendance` → `module="attendance"`
   - `/timetable` → `module="timetable"`
   - `/communication` → `module="communication"`
   - `/reports` → `module="reports"`
   - `/documents` → `module="documents"`
   - `/workflows` → `module="workflows"`

**File**: `src/providers/AuthProvider.tsx`

**Changes**:
1. **Fix ROLE_DASHBOARD_MAP**: Change `student: '/dashboard'` to `student: '/student-portal'` and `parent: '/dashboard'` to `parent: '/parent-portal'`

**File**: `src/providers/ConfigProvider.tsx`

**Changes**:
1. **Add role-based navigation filtering**: Enhance `filterItems` to also filter based on role-specific module access (map roles to allowed module IDs)
2. **Add Student portal navigation items**: Define `STUDENT_NAVIGATION_ITEMS` with student-specific paths
3. **Add Parent portal navigation items**: Define `PARENT_NAVIGATION_ITEMS` with parent-specific paths
4. **Add Teacher navigation items subset**: Filter BASE_NAVIGATION_ITEMS to only show modules Teacher has access to
5. **Add Accountant navigation items subset**: Filter to only finance-related modules
6. **Add Staff navigation items subset**: Filter to limited view-only modules
7. **Add sub-navigation items**: Add children arrays to items like Fees (sub-links to structure, payments, defaulters, analytics, make-payment) filtered by action permissions

**File**: Fee module pages (`src/modules/fees/pages/*.tsx`)

**Changes**:
1. **Add usePermission checks to action buttons**: Gate Create Fee Structure with `hasPermission('fees', 'create')`, Collect Payment with `hasPermission('fees', 'collect')`, Process Refund with `hasPermission('fees', 'refund')`, Delete with `hasPermission('fees', 'delete')`

## Testing Strategy

### Validation Approach

The testing strategy follows a two-phase approach: first, surface counterexamples that demonstrate the bugs on unfixed code, then verify the fix works correctly and preserves existing behavior.

### Exploratory Bug Condition Checking

**Goal**: Surface counterexamples that demonstrate the bugs BEFORE implementing the fix. Confirm or refute the root cause analysis. If we refute, we will need to re-hypothesize.

**Test Plan**: Write tests that render the Sidebar, simulate logins for each role, and assert navigation items match expected RBAC-filtered sets. Run these tests on the UNFIXED code to observe failures.

**Test Cases**:
1. **Sidebar Navigation Test**: Render Sidebar for a Teacher role and assert it does NOT contain links to /fees, /staff, /admissions, /settings (will fail on unfixed code — all items shown)
2. **Post-Login Routing Test**: Simulate Student login and assert navigation to `/student-portal` (will fail — routes to `/dashboard`)
3. **Route Access Test**: Render `/fees/structure` as a Teacher and assert redirect to `/permission-denied` (will fail — page renders)
4. **Fee Action Button Test**: Render FeeStructurePage as a Teacher and assert "Create Fee Structure" button is not visible (will fail — button shown without permission check)
5. **Super Admin Sidebar Test**: Render Sidebar for Super Admin and assert it contains links to `/admin/tenants`, `/admin/analytics`, `/admin/subscriptions` (will fail — shows hardcoded school navigation)
6. **Tenant Context Test**: Render Sidebar and assert it does NOT contain hardcoded "Sunrise Academy" text but instead reads from tenant state (will fail — hardcoded text present)

**Expected Counterexamples**:
- Sidebar renders same 13 hardcoded items regardless of role
- Student/Parent users land on `/dashboard` instead of portal pages
- Teacher can access `/fees/structure` URL without being blocked
- Fee pages render all action buttons without permission checks
- Possible root causes confirmed: static SECTIONS array, incorrect ROLE_DASHBOARD_MAP, RouteGuard without props

### Fix Checking

**Goal**: Verify that for all inputs where the bug condition holds, the fixed functions produce the expected behavior.

**Pseudocode:**
```
FOR ALL session WHERE isBugCondition(session) DO
  navItems := ConfigProvider.generateNavigation(session.role, session.permissions, session.tenantModules)
  sidebarItems := Sidebar.render(navItems)
  
  // Sidebar matches ConfigProvider output
  ASSERT sidebarItems = navItems.items
  
  // Post-login routing correct
  ASSERT loginRedirect(session.role) = expectedDashboard(session.role)
  
  // Unauthorized routes blocked
  FOR EACH route WHERE NOT hasModuleAccess(session.role, route) DO
    ASSERT navigate(route) REDIRECTS TO /permission-denied
  END FOR
  
  // Fee buttons gated
  FOR EACH button IN feeActionButtons DO
    ASSERT button.visible = hasPermission(session.role, button.requiredPermission)
  END FOR
END FOR
```

### Preservation Checking

**Goal**: Verify that for all inputs where the bug condition does NOT hold, the fixed function produces the same result as the original function.

**Pseudocode:**
```
FOR ALL session WHERE NOT isBugCondition(session) DO
  // School Admin access preserved
  ASSERT F(session).dashboardRoute = F'(session).dashboardRoute
  ASSERT F(session).modulePages ⊆ F'(session).modulePages
  
  // Existing permission checks preserved
  ASSERT F(StudentListPage).addButtonVisibility(session) = F'(StudentListPage).addButtonVisibility(session)
  ASSERT F(StudentProfilePage).editButtonVisibility(session) = F'(StudentProfilePage).editButtonVisibility(session)
  
  // UI behaviors preserved
  ASSERT F(sidebar).collapseToggle = F'(sidebar).collapseToggle
  ASSERT F(sidebar).logout = F'(sidebar).logout
END FOR
```

**Testing Approach**: Property-based testing is recommended for preservation checking because:
- It generates many session configurations automatically across role/permission combinations
- It catches edge cases where a permission combination might incorrectly hide a previously visible item
- It provides strong guarantees that School Admin experience is unchanged

**Test Plan**: Observe behavior on UNFIXED code first for School Admin and existing permission-gated actions, then write property-based tests capturing that behavior.

**Test Cases**:
1. **School Admin Navigation Preservation**: Verify School Admin can access all currently-accessible module pages after fix
2. **Student Page Permission Preservation**: Verify StudentListPage "Add Student" button visibility logic is unchanged
3. **Logout Flow Preservation**: Verify logout redirects to /login and clears session identically
4. **Sidebar Collapse Preservation**: Verify collapse/expand toggle continues to work with icon-only mode
5. **Demo Accounts Preservation**: Verify login page demo accounts quick-fill continues to populate credentials

### Unit Tests

- Test ConfigProvider `filterItems` logic for each role (super_admin, school_admin, accountant, teacher, staff, student, parent)
- Test ROLE_DASHBOARD_MAP returns correct paths for all roles
- Test RouteGuard redirects to `/permission-denied` when role lacks module access
- Test RouteGuard allows access when role has correct permissions
- Test Sidebar renders items from props (not hardcoded)
- Test role badge color mapping for each role
- Test fee page action buttons gated by usePermission

### Property-Based Tests

- Generate random role + permission combinations and verify sidebar items are a subset of ConfigProvider output
- Generate random role + route combinations and verify unauthorized routes redirect to `/permission-denied`
- Generate random role configurations and verify School Admin always has full access to institution modules
- Generate random fee module actions and verify buttons only render when permission is granted

### Integration Tests

- Test full login flow for each role: login → correct redirect → correct sidebar → correct page access
- Test role switching (logout → login as different role) shows correct navigation
- Test direct URL access to unauthorized modules results in permission-denied page
- Test Super Admin can navigate to all platform management pages via sidebar
- Test Student/Parent portals show only self-service navigation and own data
